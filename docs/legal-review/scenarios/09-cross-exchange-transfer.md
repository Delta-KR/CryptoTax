# 시나리오 #9 — 거래소 간 이동 (transfer 처리)

| 항목 | 값 |
|------|---|
| **검증 포인트** | 거래소 간 가상자산 이동은 양도가 아니므로 비과세 + 거주자별 통합 시 매수↔매도 자연 매칭 |
| **적용 법령** | 소득세법 시행령 §88① (거주자별 통합) · §88 (양도 정의) |
| **난이도** | ⭐⭐⭐⭐ |
| **과세 연도** | 2027년 |
| **사용자 유형** | 거주자 (개인) |
| **거래소** | Upbit (매수) → Bithumb (매도) |
| **자산** | BTC 단일 |

> Kontaxt 의 데이터 모델은 BUY/SELL 2종만 다룹니다. 거래소 간 출금·입금 (transfer) 은 매수/매도가 아니므로 CSV 업로드 시 자동으로 제외됩니다. 시행령 §88① 의 거주자별 통합에 따라 Upbit 의 매수와 Bithumb 의 매도가 자연스럽게 매칭되어 정상 처리됩니다.
>
> **⚠ 형식 한계 명시**: Bithumb XLS 파서는 [TASKS.md Phase 8](../../TASKS.md) 에서 구현 예정입니다. 본 시나리오의 Bithumb 거래는 **거래소 간 이동 + 거주자별 통합 알고리즘 검증** 목적이며, 거래소 이름과 무관하게 엔진 동작이 같으므로 알고리즘 정합성 검증에는 영향이 없습니다. 실제 Bithumb XLS 파일 업로드는 Phase 8 완료 후 가능합니다.

## 2. Kontaxt 계산 결과 (기대값)

### 2-1. 거주자별 통합 처리

| 항목 | 값 |
|------|---:|
| 총 매수 부대비용 포함 가액 (Upbit) | 5,005,000 KRW |
| 총 매수 수량 | 0.10 BTC |
| **총평균단가** | **50,050,000 KRW/BTC** |

거래소 무관 통합되므로, Upbit 의 매수와 Bithumb 의 매도가 동일 거주자의 거래로 합산됩니다.

|--------|--------|---------:|---------:|-----------:|------------:|
| 2 | Bithumb | 8,000,000 | 5,005,000 (= 0.10 × 50,050,000) | 8,000 | **+2,987,000** |

## 3. 산식 흐름

```
[STEP 0] 데이터 정제
  사용자 CSV 에 transfer (출금·입금) 라인이 있어도, Kontaxt 는 BUY/SELL 만 처리.
  → transfer 라인은 자동 제외 (양도 아니므로 과세 대상 아님)

[STEP 1] 거주자별 통합 평균단가 (시행령 §88①)
  Upbit BUY (0.10 BTC) + Bithumb BUY (0 BTC) = 통합 매수 0.10 BTC
  avg = 5,005,000 / 0.10 = 50,050,000

[STEP 2] 매도 손익 (Bithumb 매도 → Upbit 매수의 평균단가 차감)
  pnl = 8,000,000 − (0.10 × 50,050,000) − 8,000 = +2,987,000

[STEP 3] 세액
  taxable_base = max(0, 2,987,000 − 2,500,000) = 487,000
  income_tax   = 97,400
  local_tax    = 9,740
  total_tax    = 107,140
```

## 4. 법령 매핑

| 단계 | 적용 조항 | 내용 |
|------|----------|------|
| STEP 0 (transfer 비과세) | 시행령 §88 (양도 정의) | 자기 명의 지갑 간 이동은 양도가 아니므로 과세 대상 아님 |
| STEP 1 (거주자별 통합) | 시행령 §88① | "거주자별로" — 거래소 단위 무관, 사용자 단위 통합 |
| STEP 2 (자연 매칭) | 시행령 §88① + §92②4호 | 통합된 매수 평균단가가 다른 거래소 매도에 그대로 적용 |

> **검증 요청**: Kontaxt 의 "transfer 라인 무시" 처리가 시행령 §88 의 양도 정의에 부합하는지, 또는 transfer 자체를 (매수↔매도 페어로) 명시적으로 처리해야 하는지 의견 요청합니다.
