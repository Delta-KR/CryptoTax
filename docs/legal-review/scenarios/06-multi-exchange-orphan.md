# 시나리오 #6 — 다거래소 통합 + 누락 매수 (orphan 매도)

| 항목 | 값 |
|------|---|
| **검증 포인트** | 거주자별 통합 (거래소 무관) + 매수 누락 시 orphan 매도 처리 |
| **적용 법령** | 소득세법 시행령 §88① (거주자별 통합) · §64의3② |
| **난이도** | ⭐⭐⭐ |
| **과세 연도** | 2027년 |
| **사용자 유형** | 거주자 (개인) |
| **거래소** | Upbit + Bithumb (다거래소) |
| **자산** | BTC 단일 |

> 사용자가 Bithumb 의 일부 매수 거래내역을 업로드하지 못한 경우를 가정합니다. Kontaxt 는 매도 수량이 보유 잔량을 초과하는 부분을 orphan 으로 분류하여 손익 0 + 매도 수수료만 손실 처리합니다 (정직성 원칙). 누락 매수 추가 업로드 시 정상 처리됩니다.
>
> **⚠ 형식 한계 명시**: Bithumb XLS 파서는 [TASKS.md Phase 8](../../TASKS.md) 에서 구현 예정입니다. 본 시나리오의 Bithumb 거래는 **거주자별 다거래소 통합 알고리즘 검증** 목적이며, 거래소 이름과 무관하게 엔진 동작이 같으므로 알고리즘 정합성 검증에는 영향이 없습니다. 실제 Bithumb XLS 파일 업로드는 Phase 8 완료 후 가능합니다.

|---|------------|--------|------|------|-----:|-----------:|---------------:|-------------:|------|

## 3. 산식 흐름

```
[STEP 1] 거주자별 통합 (시행령 §88①)
  total_amount = 0.10 BTC (Upbit 매수만 — Bithumb 매수 없음)
  avg_price    = 5,005,000 / 0.10 = 50,050,000

[STEP 2] 매도별 손익
  매도 #2 (Upbit, 정상):
    pnl = 8,000,000 − (0.10 × 50,050,000) − 8,000 = +2,987,000
  매도 #3 (Bithumb, 잔량 0 → orphan):
    pnl = 0 + (−매도수수료) = −4,500
    + warning 출력

[STEP 3] 세액
  total_pnl    = 2,987,000 − 4,500 = 2,982,500
  taxable_base = max(0, 2,982,500 − 2,500,000) = 482,500
  income_tax   = 96,500
  local_tax    = 9,650
  total_tax    = 106,150
```

## 4. 법령 매핑

| 단계 | 적용 조항 | 내용 |
|------|----------|------|
| STEP 1 (다거래소 통합) | 시행령 §88① | "거주자별로" — 거래소 단위가 아닌 사용자 단위 통합 |
| STEP 2 (orphan 처리) | (실무적 보수 처리) | 매수 데이터 누락 시 손익 0 + 매도 수수료만 손실. 과대 신고 회피 |
| STEP 2 (warning) | (실무 안내) | 사용자에게 누락 매수 추가 업로드 또는 의제취득가액 적용 확인 요청 |
| STEP 3 (세액) | 시행령 §64의3② + 지방세법 §93 | 분리과세 20% + 250만 공제 + 지방세 10% |

> **검증 요청**: orphan 처리 (손익 0 + 매도 수수료만 손실) 가 보수적 접근인지, 또는 다른 처리 방식 (예: 의제취득가액 fallback, 또는 매도 전체 무시) 이 적절한지 의견 요청합니다.
