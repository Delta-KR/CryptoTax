# Security audit — 2026-05-23

Scope: Next.js 14 + Supabase + Resend (kontaxt). Report-only.
Methodology: code-walk of `app/`, `lib/`, `supabase/`, `middleware.ts`,
`next.config.mjs`, `package.json`. Cross-checked against PRs #28–#32.

---

## P0 (critical / exploit possible)

### P0-1 — Naver OAuth start endpoint is fully unauthenticated and unrate-limited; can be weaponised to (a) plant arbitrary `naver_oauth_state` cookies and (b) consume the project's Naver OAuth client quota / Supabase email pool

`app/api/auth/naver/start/route.ts:11-35`

The `GET /api/auth/naver/start` route is public, accepts no input, and on every
hit issues a `Set-Cookie: naver_oauth_state=<random>; SameSite=Lax; …` and a
302 to Naver. There is no rate limit, no Turnstile, no auth gate. Coupled with
`app/api/auth/naver/callback/route.ts:106-138`, which on a valid Naver callback
calls `admin.auth.admin.generateLink({ type: 'magiclink', email })` — meaning a
single successful flow can be forced repeatedly.

Reproduction sketch:
1. `for i in $(seq 1 10000); do curl -s https://kontaxt.kr/api/auth/naver/start > /dev/null; done`
2. Each call costs a Naver authorize redirect; the cookie-set is cheap but
   pollutes the user's cookie jar.
3. Worse, a Naver-attacker who can also drive a victim through Naver auth (or
   reuse a captured `code`) can spam `generateLink` on a victim's email; each
   issued link can be exchanged for a session via the published `action_link`
   redirect at `/auth/finish`. Supabase email-pool throttling and Naver code
   single-use protect somewhat, but there is no app-level cap.

Fix: Add Upstash rate limit to both `/api/auth/naver/start` and
`/api/auth/naver/callback` (IP-based, `10/min` and `5/min` respectively). Reuse
`getReportRateLimit` pattern from `lib/rate-limit.ts`.

### P0-2 — `/api/auth/naver/start` writes the CSRF `state` cookie with `path=/`, no `__Host-` prefix, and no domain pin — exploitable for cross-subdomain CSRF state fixation

`app/api/auth/naver/start/route.ts:28-34`

```ts
response.cookies.set('naver_oauth_state', state, {
  httpOnly: true,
  secure: true,
  sameSite: 'lax',
  maxAge: 600,
  path: '/',
});
```

If kontaxt is ever served from a subdomain alongside another app (e.g.
`app.kontaxt.kr` and `marketing.kontaxt.kr`) this cookie is visible to both.
A malicious sibling subdomain (or an XSS on a Vercel preview at `*.vercel.app`)
can overwrite `naver_oauth_state` to a fixed value, then trick a victim into
completing the flow with the attacker's `code` — yielding session-fixation on
the victim's account.

Fix: rename the cookie to `__Host-naver_oauth_state` (forces `path=/`, no
`domain`, `secure`) and use `sameSite: 'strict'` since the start endpoint is
hit via user navigation, not a third-party.

### P0-3 — Supabase auth client (`lib/supabase/client.ts`) is initialized without overriding default cookie options; `signInWithPassword`, `signUp`, `resetPasswordForEmail` all run with NO server-side rate limit beyond Supabase's defaults

`lib/auth/index.ts:127-187`, `lib/supabase/client.ts:1-8`

All password/OTP-bearing endpoints (`signInWithPassword`, `signUp`,
`resetPasswordForEmail`, `updateUser({password})`) run entirely through
`@supabase/ssr` browser client. Turnstile gates the first call from each
session, but a single solved token can be reused for as many in-tab calls as
the JS client wants. There is no Upstash `kontaxt-auth` window. Supabase
project-level throttling helps but is not enforced by your code.

Concretely: an attacker who solves one Turnstile (manually or via
2captcha/CapMonster) can:
- Enumerate emails via the timing/response of `resetPasswordForEmail`
  ("For security purposes, you can only request this after…" leaks an existing
  account when rate window passes).
- Brute-force the `oldPassword` check in `changePassword`
  (`app/actions/account.ts:68-87`) — `signInWithPassword` is called per
  request, no per-user lockout, just the "wrong_password" branch.

Fix:
1. Wrap the `signInWithPassword` reauth in `changePassword` with a per-userId
   rate limit (e.g. 5 attempts / 15 min).
2. Add IP rate limit to `/api/auth/naver/start` and to any
   server-action-form wrappers around `signInWithPassword`/`signUp`/
   `resetPasswordForEmail`.

### P0-4 — `app/api/auth/naver/callback/route.ts` logs `naverUser.email` in error paths and exchanges the Naver `code` *before* validating `state`-equals-cookie is constant-time — minor info-disclosure, but together with P0-1 lets an attacker probe email existence

`app/api/auth/naver/callback/route.ts:25,89-99`

`state !== cookieState` uses a normal `!==` (string compare, early-exit). Not
constant-time but acceptable since the state is 128 bits of randomness;
nevertheless, the bigger issue is on line 89-99: the callback logs the
*list of provider identities* attached to the email
(`identities.map((i) => i.provider).join(',')`). On Vercel/Supabase logs this
is fine, but it confirms that the email is registered (and via which provider)
in any environment where logs are reviewable by a third party. Combined with
P0-1 (unmetered start endpoint), an attacker who can drive Naver auth at scale
can build a provider map for any email.

Fix:
1. Log only `lookup.user.id`, never email or provider list, on the
   "registered with different provider" branch.
2. Add rate limit (see P0-1) so this branch can't be hit at scale.

## P1 (high — should fix)

### P1-1 — `dangerouslySetInnerHTML` in `app/layout.tsx:149` ships an inline boot script that runs on every page, including unauthenticated routes; the JSON-LD blob at line 152 is also `dangerouslySetInnerHTML`'d

`app/layout.tsx:67-153`

The boot script is static (no user input), so XSS via this path requires
attacker control over `app/layout.tsx` itself. **No exploit today.** However,
the CSP allows `'unsafe-inline'` precisely because of these two scripts (see
`next.config.mjs:11`). That weakens CSP defense-in-depth across the entire
site (e.g. an injected `<script>some()</script>` in any reflected-XSS sink
will execute).

Fix: migrate to CSP nonces. Next.js 14 supports nonces via middleware. Remove
`'unsafe-inline'` from `script-src`.

### P1-2 — `app/(app)/transactions/page.tsx:417` (and `lib/storage/session.ts:219`) accept arbitrary user-supplied JSON via the "import backup" file picker; zod validates schema but `saveSession()` then writes to `localStorage` keyed by `currentUserId` — a malicious JSON crafted to pass schema (e.g. 10,000 rows of bogus tax data) can be social-engineered onto a victim

`app/(app)/transactions/page.tsx:404-431`, `lib/storage/session.ts:219-242`

`importFromJSON` runs `sessionSchema.safeParse(parsed)` (good) but does not
warn the user if the imported JSON is unsigned/untrusted. The user is asked
to confirm via `window.confirm`, but the prompt does not mention "only import
files YOU exported." Combined with `saveSession()` performing
`void saveSnapshot(data)` to the **server**, a victim who imports a malicious
JSON will overwrite their server-side backup with attacker-chosen tax data.

This isn't an account-takeover but a tax-data corruption vector. For a tax
SaaS, "wrong PDF" is a serious user-impact bug.

Fix:
1. UI: stronger warning text — "백업 파일은 본인이 직접 내보낸 파일이어야 합니다."
2. Consider HMAC-signing exported JSON with a per-user secret (or just user.id
   bound) and verifying on import. Reject unsigned/foreign files.

### P1-3 — `app/api/report/route.ts:153-180` server re-computes the tax result (good!) but then **trusts `body.result.rateSource` from the client** and embeds it in the PDF as the "audit trail" of which exchange rate source was used

`app/api/report/route.ts:175-187`

```ts
const wire = resultToWire(serverResult, 'premium');
wire.rateSource = body.result.rateSource; // ← from client
```

Comment acknowledges this: "rateSource는 클라이언트가 보낸 값을 그대로 신뢰".
The dollar amount is recomputed server-side, but the *displayed source*
(`'Upbit (KRW market daily close)'` vs `'Static fallback'`) is whatever the
client says. A user could ship a PDF claiming all rates came from "DB" when
they actually came from the static fallback, weakening the PDF's value as
evidence with a tax authority.

Risk is real (this is a Korean tax product where the PDF is the deliverable)
but bounded — the actual tax numbers (`pricePerUnitKRW`) **are** validated
because they're embedded in the verified `transactions` array.

Fix: server should recompute `rateSource` from the same DB query used for
recalculation, not echo `body.result.rateSource`.

### P1-4 — `/api/report` rate limit is IP-only (not user-bound); two free-tier users behind the same NAT will share the 10/min budget; conversely, one premium user from a residential IP can hammer expensive PDF generation through different sessions until the IP-window resets

`app/api/report/route.ts:71-91`, `lib/rate-limit.ts:52`

`getReportRateLimit()` uses `Ratelimit.slidingWindow(10, '1 m')` keyed by IP.
The route already does `requirePremium`, so the budget is largely a DoS
shield, but coupling to IP only is the wrong primitive — a paying premium
user behind corporate NAT (very common in Korea) can be blocked because
someone else on the same IP is generating PDFs.

Fix: key on `user:${userId}` after auth, fall back to IP only for the brief
window before `requirePremium` returns. Already the pattern used in
`calculate.ts:53-74`.

### P1-5 — `app/auth/callback/route.ts` does not constrain `next` to be a path on the same origin beyond `safeNext()`; `safeNext` does not handle URL-encoded protocol attacks like `next=/%5cevil.com`

`lib/auth/safe-next.ts:1-10`

Regex `^\/[^/\\]/` rejects `//evil.com` and `/\evil.com` but does not decode
percent-encoded sequences before checking. `next=/%2fevil.com` becomes
`/%2fevil.com` (literal, treated as same-origin path) — but the browser may
interpret it as `//evil.com` after server-side decoding when handed to
`new URL(next, url.origin)`. Quick test:

```
new URL('/%2fevil.com', 'https://kontaxt.kr')
// → URL { href: 'https://kontaxt.kr/%2fevil.com', pathname: '/%2fevil.com' }
```

This actually stays on-origin in WHATWG URL — the percent-encoding is
preserved as a path, not interpreted as scheme. So **NO open-redirect today**,
but the safety relies entirely on `URL`'s normalization. Belt-and-braces:
decode then re-check.

Fix: add `try { decoded = decodeURIComponent(next); if (!/^\/[^/\\]/.test(decoded)) return DEFAULT_NEXT } catch { return DEFAULT_NEXT }`.

### P1-6 — `/auth/finish` (`app/auth/finish/page.tsx`) blindly calls `supabase.auth.setSession()` with `access_token` + `refresh_token` from the URL fragment — any page on kontaxt origin that later sets `window.location.hash` can hijack a fresh session

`app/auth/finish/page.tsx:20-37`

Fragment-based token delivery is standard (Supabase magic-link/implicit
flow), but the `/auth/finish` component does not verify that the page was
*arrived at* via redirect from `/api/auth/naver/callback` (no `state`, no
referrer check, no nonce). An attacker who can get a victim to visit
`https://kontaxt.kr/auth/finish#access_token=ATTACKER_TOKEN&refresh_token=…`
(e.g. via a phishing email link) will sign the victim **in as the attacker**,
then any data the victim enters (transactions, profile updates) lands on the
attacker's account, which the attacker can then read.

Severity is medium-high — requires phishing, but easy to weaponize.

Fix: bind `/auth/finish` to a nonce stored in cookie at start; verify on
finish before calling `setSession`. Or require the user to click a confirm
button after presenting `data.user.email` from the implicit session.

### P1-7 — Supabase Edge function `fetch-daily-rates` is `verify_jwt=false` (inferred from comments) and protected only by a shared secret in the `x-shared-secret` header; the secret is loaded from Supabase Vault but the migration `20260522030000_cron_shared_secret.sql` falls back to empty string if Vault is missing, silently disabling auth on cron

`supabase/migrations/20260522030000_cron_shared_secret.sql:31-39`

```sql
if v_secret is null then
  raise warning '...';
  v_secret := '';
end if;
```

If the Vault entry is dropped or the migration is re-run on a fresh project,
the cron schedule will send `x-shared-secret: ` (empty). The edge function
**correctly rejects** that with 401 (`fetch-daily-rates/index.ts:87-93`), so
cron silently fails. But this is configuration-fragile — a misapplied
migration disables your DB freshness pipeline without anyone noticing until
calculations come back with stale rates.

Fix: `raise exception` instead of `raise warning` when Vault is missing. Fail
the migration loudly so prod can't enter this state.

### P1-8 — `sessionSchema.allParsed: z.array(z.record(z.unknown())).max(50_000)` permits arbitrary keys/values in `localStorage` session data; the server `saveSnapshot` route re-validates with the same schema but again with `z.unknown()` for `allParsed` items — server stores attacker-controlled JSON in `user_data.payload`

`lib/storage/session.ts:67-85`, `app/actions/user-data.ts:35-65`

Each user's session payload (up to 50,000 records × `z.record(z.unknown())`)
is stored in `user_data.payload` (JSONB). RLS confines it to the owner, so
the worst case is the user storing junk in their own row. **No
cross-tenant leak.** But Supabase JSONB row limit is 1GB, and 50k records of
unbounded shape could become large — DoS via storage.

Fix: tighten `allParsed` to use `parsedTransactionSchema` from
`lib/validation/calculate.ts` (it already exists). The current `z.unknown()`
is a leftover that should be replaced.

## P2 (medium)

### P2-1 — `next.config.mjs:11` includes `'unsafe-inline'` in `script-src` and `'unsafe-inline'` in `style-src` for the entirety of production; with two known inline scripts only, this is a real CSP regression risk

`next.config.mjs:8-26`

Comment acknowledges this is for `themeRestoreScript`, JSON-LD, and Tailwind
inline styles. Migration plan ("nonce 도입 시 별도 PR에서 제거") is noted but
not executed. Until then, any reflected XSS sink anywhere on the site is
fully exploitable.

Fix: use Next.js middleware nonces (`generateCSP(nonce)`). See
https://nextjs.org/docs/app/building-your-application/configuring/content-security-policy.

### P2-2 — `app/api/report/route.ts:34-58` `isAllowedOrigin` allows any `*.vercel.app` host when `process.env.VERCEL` is set — preview environments can be used by external attackers if they discover a kontaxt preview URL

`app/api/report/route.ts:48-52`

Vercel preview deployments are sometimes searchable, sometimes shared. The
preview environment shares the production Supabase project (unless explicit
branching). An attacker who finds a preview URL can make `POST /api/report`
calls that bypass the origin check.

Fix: gate `isAllowedOrigin` Vercel branch to `process.env.VERCEL_ENV ===
'preview'` *and* additionally require a known-prefix path on `*.vercel.app`,
or just disable preview origin allowance entirely if previews share prod
Supabase.

### P2-3 — `app/actions/upgrade-plan.ts` is hard-disabled (good!) but the comment promises a future "service_role 기반으로 복구" — leaving a footgun for whoever lands the Toss webhook PR

`app/actions/upgrade-plan.ts:1-21`

Pre-emptive note: when Toss is wired in, the server action MUST verify the
Toss webhook signature against `TOSS_WEBHOOK_SECRET` **before** any
`updateUser` / `profiles.update` call. Currently no `lib/billing/` skeleton
exists, so reviewer-flag this for the next PR.

### P2-4 — `app/actions/account.ts:171-176` `redirect('/')` after `admin.deleteUser` runs *after* the user is deleted; if `cookies()` is still readable, the cookie-store mutation in `signOut()` (line 163) is best-effort — orphaned auth cookies linger on disk

`app/actions/account.ts:160-177`

Minor. After deletion, the auth cookies are invalid (server-side check
fails), so the worst case is one extra page load where `getUser()` returns
null and middleware redirects to `/login`. Acceptable.

Fix (optional): use `cookies().delete()` for all `sb-*` cookies before
`redirect`.

### P2-5 — `lib/parsers/upbit.parser.ts:175-185` dynamically imports `pdf-parse` at runtime; pdf-parse 1.1.4 has a known prototype-pollution / large-PDF DoS history. CSV/PDF size cap is 10MB which limits memory blowup, but a malicious 10MB PDF can still trigger long parse loops (no per-file timeout)

`lib/parsers/upbit.parser.ts:163-216`

Mitigated by:
1. `MAX_FILE_BYTES = 10 * 1024 * 1024` in `lib/validation/calculate.ts:4`.
2. Rate limit of 20 calls/min/user.
3. `pdf-parse` runs in the Node.js runtime with a 60s `maxDuration` on the
   action.

Fix: wrap `pdf(buf)` in `Promise.race([pdf(buf), timeout(30_000)])` to bail
out on adversarial PDFs.

### P2-6 — Edge function input validation accepts arbitrary date back to `2022-01-01` which can backfill huge ranges; combined with `coins` up to 50, an authenticated attacker (with the shared secret) could trigger a 50*1500-day Upbit fetch loop. Auth gate (shared secret) protects, but if leaked, abuse is easy.

`supabase/functions/fetch-daily-rates/index.ts:27, 155`

Acceptable since `FETCH_RATES_SHARED_SECRET` is server-only. Worth noting:
the secret rotation procedure exists (see `supabase/README.md`) but no
documented rotation cadence.

### P2-7 — `app/(app)/transactions/page.tsx:417` shows `result.transactionCount` from `importFromJSON` directly back to the user; while the data is sanitized through Zod, the error message at line 425 echoes Zod's raw error path. Low-priority but worth tightening to avoid disclosing schema internals.

`lib/storage/session.ts:230-234`

Not exploitable, hygiene improvement.

### P2-8 — `app/auth/callback/route.ts:50` triggers `sendWelcomeEmail()` based on `user.created_at === user.last_sign_in_at` heuristic — if `last_sign_in_at` is not updated atomically by Supabase, the welcome email can be sent on every login for some users

`app/auth/callback/route.ts:50-61`

Minor user-facing bug + Resend cost. Confirm with Supabase docs that
`last_sign_in_at` is set before `exchangeCodeForSession` returns.

## P3 (low / hygiene)

Not security-relevant (IDs are local UI only), but flagged as hygiene — use
`crypto.randomUUID()` for consistency.

Pure cosmetic, safe.

Add tests or use error codes (newer Supabase exposes `error.code`).

Defensive.

### P3-7 — `app/(app)/transactions/page.tsx:407-413` uses `window.confirm()` for the import-overwrite warning. `confirm()` UX is poor and easy to dismiss. Replace with the existing `Modal` component.

### P3-9 — No `Cross-Origin-Opener-Policy` / `Cross-Origin-Embedder-Policy` headers. Not strictly required, but OAuth popups (Supabase signInWithOAuth) benefit from `same-origin-allow-popups`.

### P3-10 — `npm audit` not run (offline). Eyeballed:

### P3-11 — Cron migration `20260521120000_setup_daily_rates_cron.sql` is *superseded* by `20260522030000_cron_shared_secret.sql` but the original file is still in the migrations folder. Both `cron.unschedule('fetch-daily-rates')` blocks run on fresh init, but if a future migration is squashed, the unauthenticated cron from the older file could re-run. Document this dependency in a migration README.

### P3-12 — `app/(auth)/signup/page.tsx:87` writes pending plan to `sessionStorage` without prefix/version; if Toss integration lands and reads this without validation, a malicious site (via supercookie) could pre-set "annual" or "subscription" to game pricing.
Fix: validate `pendingPlan` server-side on checkout, not just on read.

## Verified clean

These areas were checked and look solid:

- **RLS policies** on `profiles`, `user_data`, `user_deemed_cost_overrides`,
  `user_imputed_expense_coins`, `daily_rates`, `deemed_cost_snapshots`: all
  enforce `auth.uid() = user_id` for SELECT/INSERT/UPDATE (and DELETE where
  applicable). The P0 hotfix `20260523010000_p0_hotfix.sql` correctly added
  `WITH CHECK` to `user_deemed_cost_overrides` UPDATE policy. No tenant
  isolation gaps in the schema.
- **service_role key handling**: confined to `lib/supabase/server.ts:29-52`
  (`createSupabaseAdminClient`) and `lib/supabase/admin-lookup.ts:31` — both
  server-only modules; `'use server'` directive is in callers; key is never
  exposed in client bundles. Admin client has `persistSession: false` and
  empty cookie handlers (good).
- **localStorage isolation (PR #28)**: `lib/storage/session.ts:30-33`
  correctly scopes keys to `kontaxt-session-v1-${userId}`; legacy single-key
  is purged on every load (`purgeLegacyKey`); `clearAllSessions` runs on
  logout. Verified.
- **Password change captcha (PR #29)**: `app/actions/account.ts:68-87`
  forwards `captchaToken` to `signInWithPassword`; the test file
  `app/actions/__tests__/account.test.ts:170-193` confirms behavior; mapping
  of "captcha protection" Supabase error to `captcha_failed` code is correct.
- **JSON backup user binding (PR #30/31/32)**: `saveSnapshot` server action
  derives `user_id` from `supabase.auth.getUser()` (`app/actions/user-data.ts:46`),
  NOT from any client param. `upsert` with `onConflict: 'user_id'` is safe.
  RLS doubly enforces. The session schema is the same on client and server,
  so a malicious client can't smuggle extra fields. **The previously-feared
  cross-user leak path is closed.**
- **CSRF in Naver OAuth state**: state cookie comparison is correct
  (`app/api/auth/naver/callback/route.ts:25`), and cookie is `httpOnly: true`,
  `secure: true`, `sameSite: 'lax'`, 10-min TTL. (Caveat: see P0-2 re:
  `__Host-` prefix.)
- **CSP frame-ancestors 'none'** + **`X-Frame-Options: DENY`** — clickjacking
  fully blocked.
- **Account takeover via OAuth merge (C2)**: `findUserByEmail` +
  `provider='naver'` identity check in the callback is correct.
  Already-registered-with-different-provider returns `?error=already_registered_other_provider`
  *before* `generateLink`. Fail-closed on REST lookup failure. Verified.
- **PDF generation re-validates tax math server-side (C5)**:
  `app/api/report/route.ts:153-173` re-computes from
  `body.transactions` after Zod validation; client-provided `result` is
  ignored for monetary values. Excellent.
- **Premium gating**: `requirePremium` in `lib/auth/server.ts:68-102` checks
  both `plan === 'premium'` AND `premium_until > now()`; client mirror in
  `lib/auth/index.ts:38-44` matches; PDF route, deemed-cost actions, and
  imputed-expense actions all gate.
- **File parsing**: Zod schema + 10MB/file, 50MB/total, 20-file caps in
  `lib/validation/calculate.ts:4-7`. MIME + extension allowlist. Both Upbit
  PDF (`looksLikeUpbitTransactionPdf`) and Binance Spot CSV (header equality)
  have signature checks; Binance Futures and Order History files are
  explicitly rejected.
- **No raw SQL in app code**: all DB access goes through Supabase client.
- **No `eval`, `new Function`, or `innerHTML` in app code** (only the two
  vetted `dangerouslySetInnerHTML` in `app/layout.tsx`).
- **No committed `.env*` files**; `.gitignore` covers `.env*.local`.
- **`SUPABASE_SERVICE_ROLE_KEY` not in client bundle** (only used in
  `lib/supabase/server.ts` and `lib/supabase/admin-lookup.ts`).
- **Login `next` redirect**: `safeNext` correctly rejects `//evil.com` and
  `/\evil.com`. (P1-5 is a defense-in-depth nit, not an actual bypass.)
- **Resident-only `totalAverage` PDF gate** at
  `app/api/report/route.ts:141-150` correctly blocks FIFO/MA PDF
  generation per §88①.
