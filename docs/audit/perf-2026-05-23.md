# Performance audit — 2026-05-23

Scope: Next.js 14.2 + React 18 + Supabase + Tailwind 3 (kontaxt). Code-walk only;
no build measurement (node_modules absent in worktree). Cross-checked against
Supabase advisor (performance) live API.

Methodology: read every page under `app/`, every component under `components/`,
all server actions, the PDF route, parser registry, rate provider, and the
session storage layer. Looked for `'use client'` overuse, dynamic imports,
waterfalls, re-render fan-out, bundle bloat, hydration issues, image/font
handling, and DB index hygiene.

Summary verdict: solid foundation — RSC vs client split is mostly clean, heavy
deps (`@react-pdf/renderer`, `pdf-parse`, `papaparse`) are correctly server-only,
fonts are self-hosted via `next/font/local`, Tailwind purge is tight, and the
DB rate provider preloads to avoid N+1. The main wins live in the App layout
(duplicate auth round-trips) and the marketing nav (one `cookies()` call drags
the whole marketing tree out of static).

---

## P0 (user-visible perf problem)

None. No render-blocking script, no client-bundled PDF/CSV libs, no obvious
runtime hot loop. Initial app HTML for `(app)/*` is gated by an inline
`useCurrentUser` loading state (`return null` until profile arrives) but that's
correctness/UX, not perf.

## P1 (notable regression risk)

### P1-1 — `useCurrentUser()` runs in `AppShell` *and* in 5 child pages independently — every page mount does ≥2 redundant Supabase calls
`lib/auth/index.ts:56-125`, `components/app-chrome/AppShell.tsx:12`,
`app/(app)/billing/page.tsx:12`, `app/(app)/report/page.tsx:30`,
`app/(app)/tax/page.tsx:518`, `app/(app)/tax/deemed-cost/page.tsx:86`,
`app/(app)/settings/profile/page.tsx:27`

Each call subscribes its own `onAuthStateChange` listener and fires its own
`supabase.from('profiles').select('plan, premium_until')` + a `loadSnapshot()`
server action (which itself does another `getUser()` + `user_data.select`).
On `/tax` for a logged-in user this fans out to:
- AppShell: 1× profiles select + 1× loadSnapshot (server action: 1× getUser + 1× user_data select)
- TaxPage: 1× profiles select + 1× loadSnapshot (server action: 1× getUser + 1× user_data select)

= 2 client-side profile selects + 2 server actions doing 4 supabase calls each
mount. The two listeners also both trigger on every auth event (token refresh,
storage sync), so navigation between (app) pages re-fires the snapshot dance.

Impact: ~200–500ms wasted on cold transitions, doubled idle Supabase load,
race conditions between the two `saveSession` calls that can write stale
snapshots over fresh ones.

Fix: lift `useCurrentUser` into a React context provider mounted once in
`AppShell` and have child pages consume the context (`useUserContext()` returns
the same `{user, loading}`). The `loadSnapshot` side-effect should also move
out of the hook into a one-shot provider effect, gated by a "did we already
hydrate this session" ref so navigation doesn't re-fetch.

### P1-2 — Marketing tree is dynamic-rendered because `Nav` calls `cookies()` for auth state
`components/sections/nav.tsx:15-20`, `lib/supabase/server.ts:5`,
`app/(marketing)/layout.tsx:1-12`

`Nav` is an async RSC that does `supabase.auth.getUser()` to swap between
"로그인 / 무료 시작" and "대시보드" CTAs. The supabase server client reads
`cookies()`, which opts the entire marketing layout out of static rendering.
`/` already has `revalidate = 3600` (so it's ISR), but `/guide`, `/sample`,
`/legal/privacy`, `/legal/terms` have no `revalidate` and therefore render on
every request with a cookie read + (probably cached) auth.getUser() roundtrip.

Impact: each marketing pageview pays ~30–80ms for a cookie/JWT decode on the
edge, and `/sample`/`/guide`/`/legal/*` cannot be served from the CDN as
static HTML. SEO crawlers and uncached cold hits feel this most.

Fix options, pick one:
- **Cheapest:** add `export const revalidate = 86400` to the legal/sample/guide
  pages — they don't change. Nav stays dynamic but per-route caching kicks in.
- **Best:** convert `Nav` to a client component that reads auth state via
  `useCurrentUser` (or a tiny `useEffect` getSession), and render the marketing
  layout fully static. Costs a one-frame "로그인" → "대시보드" flicker for
  logged-in users on marketing pages — usually fine.

### P1-3 — `/api/report` does the auth round-trip twice
`app/api/report/route.ts:93-114`, `lib/auth/server.ts:68-102`

`requirePremium()` already calls `supabase.auth.getUser()` *and* selects from
`profiles`. The route then calls `supabase.auth.getUser()` a second time just
to grab `user.email` / `user.user_metadata.name` for the filename. PDF
generation is already 2–5s — an extra 50–150ms auth roundtrip is small but
unforced.

Fix: have `requirePremium` return the `user` object (it has it; just add to
`PremiumGuardOk`). The second `getUser()` at line 107-114 becomes one property
read.

### P1-4 — `lib/mock/tax.ts` `calculateTax()` parses the entire localStorage blob through Zod on every call, every render
`lib/mock/tax.ts:116-201`, `lib/storage/session.ts:87-109`,
`app/(app)/tax/page.tsx:530`, `app/(app)/dashboard/page.tsx:102-123`,
`app/(app)/report/page.tsx:47-50`

`calculateTax()` is wrapped in `useMemo`, but its deps include `txs` (a freshly
allocated array each render via `getTransactions()`) on tax/dashboard/report.
More importantly, inside it calls `loadSession()` → `localStorage.getItem` →
`JSON.parse` → `sessionSchema.safeParse` (a non-trivial Zod schema with a
50,000-element `z.array(...)` upper bound). For a user with 1k transactions
this is multi-millisecond work fired on every keystroke in a filter or year
toggle.

The tax page's `useMemo` also depends on a `refreshKey` ref that's incremented
after a server recalc, so the memo busts even when txs/method/year haven't
moved. Combined with the duplicate `useCurrentUser`, the page does noticeable
work on each year/method toggle.

Fix: store the parsed `SessionData` in a top-level cache (module-level WeakMap
keyed by user id, or simply a memoized result keyed by `localStorage` mtime)
and return the same reference until invalidated. Even just caching the JSON
parse + zod result on first read saves ~3ms per render.

### P1-5 — `<img>` on `mark.svg` instead of `next/image` in the nav/footer, with no fixed width
`components/sections/nav.tsx:33`, `components/sections/footer.tsx:46`

`<img src="/mark.svg" className="h-8 w-8" />` — `w/h` set via Tailwind only.
Browser doesn't know dimensions until the SVG parses, so this is a small but
real CLS contributor on first paint of every marketing page. The
`/* eslint-disable @next/next/no-img-element */` comment owns the choice, but
the rationale ("mark.svg 1500×1500 viewBox") doesn't apply to CLS — adding
`width="32" height="32"` attributes (or switching to `<Image fill />`) costs
nothing.

Fix: replace with `<Image src="/mark.svg" alt="" width={32} height={32}
className="h-8 w-8 shrink-0" priority />` in nav (priority because it's
above-the-fold on every marketing page).

### P1-6 — `doge.svg` is 56 KB — ~30× bigger than the other coin SVGs
`public/logos/coins/doge.svg` (57184 bytes), compare `btc.svg` 2 KB,
`eth.svg` 1.4 KB, `sol.svg` 1.8 KB, `xrp.svg` 0.5 KB

Rendered through `<Image>` in `CoinIcon`, which doesn't optimize SVG content.
On a dashboard listing many DOGE rows this is fine (one cached fetch), but
for first-paint of any page that shows DOGE it's a ~55KB blocker. Visual
inspection of `doge.svg` likely shows un-optimized raster traces inside the
SVG.

Fix: run through SVGO or hand-clean: target ≤4 KB like the others. Saves
~52 KB on any first paint involving DOGE.

### P1-7 — `<Image>` used for tiny exchange logos that are PNGs in `public/logos/*.png`
`components/sections/hero.tsx:139-145`, `components/sections/problem.tsx:58-63`,
`components/sections/exchanges.tsx`, `app/(app)/transactions/upload/page.tsx:217`

PNG files like `upbit.png` (1.6 KB), `binance.png` (0.9 KB), `bybit.png` (0.7 KB),
`gate.png` (11 KB) are served through `next/image`. The Vercel image
optimizer adds latency + cache fan-out for files this small; for a 700-byte
PNG the optimizer round trip is slower than the raw file. `gate.png` at 11 KB
also looks suspicious for a 22×22 logo.

Fix: either (a) inline-import these as static imports so Next can skip the
optimizer for already-optimal sources, (b) replace `binance.png` / `bybit.png`
/ `upbit.png` with SVGs (cheap, brand colors are flat), or (c) re-export
`gate.png` at the actual rendered size.

### P1-8 — `uuid` v14 imported in 5 hot server modules; could be `crypto.randomUUID()`
`lib/engine/tax-calculator.ts:1`, `lib/engine/total-average.ts:1`,
`lib/engine/normalizer.ts:1`, `lib/parsers/upbit.parser.ts:1`,
`lib/parsers/binance-spot.parser.ts:2`

`uuid@^14` pulls in ESM-only entry points and a polyfill layer; Node 19+ has
`globalThis.crypto.randomUUID()` natively (Vercel runtime is Node 20). All
five sites use `v4 as uuid` and only need v4. Switching saves ~10–20 KB on
the server bundle and a tiny amount of startup time.

Fix: `const uuid = () => crypto.randomUUID()` helper in `lib/engine/ids.ts`,
swap five imports. Drop `uuid` + `@types/uuid` from `package.json`.

### P1-9 — Supabase advisor: 10 RLS policies still re-evaluate `auth.uid()` per row
Live `mcp__supabase__get_advisors` returned 10× `auth_rls_initplan` WARN on
`user_deemed_cost_overrides` (select/insert/update/delete), `user_imputed_expense_coins`
(select/insert/delete), and `user_data` (select/insert/update).

Migration `supabase/migrations/20260523030000_optimize_rls_initplan.sql` is in
the repo and rewrites all 10 policies to `(select auth.uid()) = user_id`, but
the advisor result confirms it hasn't been applied to the live DB yet (caller
note: "separately tracked"). Once applied, these 10 entries clear and the
`user_data` table — which gets a `select` on every (app) page mount via
`loadSnapshot()` — stops paying the per-row initplan cost.

Fix: apply the migration to prod. No code change required.

## P2 (polish)

### P2-1 — `app/layout.tsx` inline boot script does a one-time localStorage migration that's already shipped
`app/layout.tsx:67-107`

~40 lines of inline JS runs *before paint on every page*, on every visit,
forever — even though the work behind `'kontaxt-migration-v1'` flag is a
no-op after the first hit. Doesn't block much (it's synchronous but small),
but it's permanently parsed and held in the HTML response, contributing ~700
bytes to every Next.js page. After 30 days, plan to delete the migration
block entirely (keep only the theme restore).

Fix (later, not now): set a calendar reminder for ~2026-07-01 to delete the
`L`/`S` migration arrays and shrink the script to the 4-line theme restore.

### P2-2 — No `<Suspense>` boundaries and no `loading.tsx`
`app/**`

The (app) pages all call `useEffect` to hydrate data from localStorage with a
spinner-less blank render until then. No `app/(app)/loading.tsx`. Navigation
between (app) pages shows a stark blank → content transition because the
server layout's auth check happens *before* React renders, and the client
hook then re-checks. Adding `loading.tsx` per route group gives Next's
built-in streaming/skeleton path.

Fix: drop a `loading.tsx` in `app/(app)/` with a thin skeleton (the
`Skeleton` component is already built). Free UX upgrade.

### P2-3 — `ThemeToggle` icon swap waits for `mounted = true`, hiding the icon for one paint
`components/theme-toggle.tsx:7-22`

Standard pattern, but the inline boot script in `layout.tsx` already sets
`data-theme` *before paint*. We could read `document.documentElement.dataset.theme`
in `useState` initializer (since this is `'use client'`) and skip the mounted
gate, avoiding the brief icon flicker.

### P2-4 — `Button.tsx` has `'use client'` but uses no client APIs
`components/ui/Button.tsx:1-3`

It's just `forwardRef` + tailwind class composition. Could be RSC — saves a
client-component-boundary marker per usage site. However, every Button caller
in this codebase is already inside a client tree (forms, toolbars), so the
practical bundle savings are zero. Cosmetic.

### P2-5 — Two unused logo files in `public/`
`public/logo.svg` (14.6 KB), `public/logo-dark.svg` (14.6 KB),
`public/kontaxt-logo.png` (16.6 KB), `public/kontaxt-logo-dark.png` (11.4 KB)

`grep` finds zero references in app code. Only `emails/components/kontaxt-logo.ts`
references `https://kontaxt.kr/kontaxt-logo-brand.png` (which *is* used). Other
four appear to be unused leftovers from the rebrand.

Fix: verify and delete.

### P2-6 — Two dormant email send functions kept in `lib/email/send.ts`
`lib/email/send.ts:97-123`

`sendVerifyEmail` and `sendResetPasswordEmail` are exported but never called
(comment confirms Supabase Custom SMTP handles those). They import full
React Email components — but since everything stays on the server, this is
zero client-bundle cost. Keep or delete is taste; flagging only for
cleanliness.

### P2-7 — `recalcTriggered` ref in `app/(app)/tax/page.tsx` only fires once per session
`app/(app)/tax/page.tsx:523, 535-556`

If a premium user lands on `/tax` with a masked free-tier result still in
localStorage (e.g., they upgraded in another tab), the auto-recalc trigger
fires once. Correct. But navigating away and back resets the ref and re-fires
the server action with the same parsed payload. Cheap fix: track in a
useEffect that depends on `user.id`, gate by `session.allUnified.some(t =>
t.rateMeta?.source === 'static' || result.masked)`. Saves the occasional
unneeded recompute on premium users.

### P2-8 — `getTransactions()` builds and sorts the full unified list on every call
`lib/mock/transactions.ts:62-93`, called from useEffect on dashboard / tax /
transactions / report pages

Each call: `loadSession()` (parse + zod), `.map` + `.filter` + `.sort` over
all unified transactions. For 5,000 transactions this is ~3–5ms. Not a P1
because it runs once per mount, but each `useCurrentUser` cascade triggers a
fresh `getTransactions()` call too. Pairs naturally with the P1-4 fix
(cache parsed SessionData at module level).

### P2-9 — `app/(marketing)/page.tsx` re-renders 11 large RSC sections on every ISR boundary
`app/(marketing)/page.tsx:17-33`

Not actually a problem — each section is a pure component with deterministic
output. Flagging only because the Hero's `DashboardMock` is 254 lines of JSX
that runs on every ISR rebuild. If you ever pre-compute parts to JSON, do it
here. (No action.)

### P2-10 — `LineChart` not used anywhere in app code
`components/ui/Chart/LineChart.tsx`

`grep` finds zero callers outside the file itself. Pulls in 280 lines of
client JS for nothing. Either wire it into a place that needs it, or delete.
Saves ~6 KB gzipped from any client chunk that incidentally imports it (none
currently, so no live impact).

## Verified clean

- **`@react-pdf/renderer`** — imported only in `app/api/report/route.ts` and
  `lib/report/tax-report.tsx`. Never reaches the client bundle.
- **`pdf-parse`** — `await import('pdf-parse')` dynamic, also listed in
  `next.config.mjs:42 serverComponentsExternalPackages`. Server-only.
- **`papaparse`** — used only in `lib/parsers/binance-spot.parser.ts`, which is
  reached only through `app/actions/calculate.ts` (`'use server'`).
- **Fonts** — `Pretendard` is self-hosted via `next/font/local` pointing to
  `node_modules/pretendard/.../PretendardVariable.woff2`; JetBrains Mono via
  `next/font/google` (auto self-host at build). `outputFileTracingIncludes`
  in `next.config.mjs:43-47` pulls the Pretendard woff2 into the `/api/report`
  bundle so PDF rendering doesn't break.
- **Tailwind purge** — `content: ['./app/**/*.{ts,tsx}', './components/**/*.{ts,tsx}']`
  is correctly scoped (no `**/*` globbing into node_modules, no missing dirs).
- **Rate provider** — `DBExchangeRateProvider.preload()` issues a single
  `IN (...)` query for all (date × from_currency) combos before normalization.
  No N+1.
- **Resolvers** — `resolveDeemedCostPrices` + `resolveImputedExpenseCoins`
  fired in `Promise.all` (`app/api/report/route.ts:162-165`,
  `lib/engine/resolvers.ts:38-45`). Parallelized.
- **Edge function `fetch-daily-rates`** — proper rate limiting via
  `COIN_DELAY_MS=1500` + `PAGE_DELAY_MS=300`, exponential backoff on 429,
  shared-secret auth, input clamping for coins/startDate.
- **CSP** — `next.config.mjs` correctly drops `'unsafe-eval'` in production.
- **Middleware matcher** — narrowly scoped to protected routes only; `/`,
  `/guide`, `/legal/*` skip the supabase middleware entirely.
- **Image optimization** — `next/image` used everywhere except the two
  intentional `<img>` tags noted in P1-5. CoinIcon falls back to a styled
  letter-circle for unknown coins (no broken images).
- **Suspense bailout risk** — no `useSearchParams()` calls outside of
  `useEffect` reading `window.location.search`. No CSR bailout.
- **`useMemo` / `useCallback`** — 30 instances total across the codebase,
  all reasonable. No memo-everywhere antipattern, no obvious missing memos
  on top-level pages.
- **Inline functions as props** — present (e.g., Select `onChange={(e) =>
  setExchange(...)}`), but child components are mostly memoized DOM with
  cheap renders. No measurable cost.
- **List `key` prop** — all `.map()` renderings checked use stable keys
  (`tx.id`, `c.coin`, `h.coin|lot.id`, etc.).
- **Pretendard subset** — uses the variable font (single file, 45–920
  weight range) — no multiple weights pulled separately.
- **Test suite** — 198 tests in ~1s is genuinely fast; spot-check of
  `lib/__tests__/e2e.test.ts` (550 lines, multi-scenario) shows no I/O,
  no real Supabase calls. Vitest config (`vitest.config.ts`) uses
  `vite-tsconfig-paths` for resolution — minimal startup overhead.
- **Tailwind config** — fontFamily uses CSS variable indirection
  (`var(--font-pretendard)`) which lets the local font load asynchronously
  without re-parsing tailwind classes.
